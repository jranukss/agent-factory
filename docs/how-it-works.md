# How the Agent Factory Works

*Architecture and concepts. For day-to-day usage, see [user-guide.md](user-guide.md).*

---

## 1. What it is

The agent factory is a **software-delivery pipeline for Claude Code**: a
single command (`/feature`) drives a work item вҖ” a *ticket* вҖ” from a one-line
request to a merged pull request, through a team of specialist AI agents with
a human approving the decisions that matter.

It is built on four principles:

1. **Files are the only handoff.** Agents run in isolated contexts and share
   nothing except the markdown artifacts they read and write in the ticket
   folder. This makes every run inspectable, auditable, and resumable.
2. **The factory never contradicts the project.** Agents carry role and
   process only. Every project-specific rule comes from the project's own
   standards doc (usually `CLAUDE.md`) and one config file, read at runtime.
3. **Human control where it's cheap, autonomy where it's safe.** Gates guard
   requirements and hard-to-reverse decisions. Everything reversible is
   decided by agents and *recorded* so you can audit it later.
4. **Ceremony scales with risk.** Three lanes (Full / Lite / Hotfix) and a
   per-gate interactivity policy mean a typo fix doesn't pay the process cost
   of a new feature.

## 2. The three layers of truth

| Layer | File | Owns | Edited by |
|---|---|---|---|
| **Protocol** | `.claude/factory/protocol.md` | The generic contract: phases, gates, loops, return codes. Identical in every project. | The factory repo (upstream) only |
| **Config** | `.claude/factory/config.md` | This project's operations: commands, paths, models, gate policy, lanes. | You (generated by `/factory-init`) |
| **Standards** | `CLAUDE.md` + docs it links | This project's coding conventions вҖ” what "correct" means here. | You (the post-merge learning gate proposes amendments; you approve) |

Precedence: your live instructions > standards (for conventions) > config
(for commands/paths). Agents read protocol + config always, and standards
*scoped to their role* via the config's role-map вҖ” except the code-reviewer,
which always reads the standards in full.

## 3. The team

| Agent | Model | Reads | Writes | Never does |
|---|---|---|---|---|
| **product-owner** | sonnet | task, product surface | `00-intake.md`, `01-spec.md`, `ROADMAP.md` | name files, schemas, or tech approach |
| **ux-designer** | opus | spec, component vocabulary | `design/mockup.html`, `02-design.md` | write app code; publish Artifacts |
| **tech-lead** | opus | spec, design, the actual code | `03-plan.md`, `standards-digest.md` | write app code; open branches |
| **senior-dev** | opus | plan, digest | **the code + tests**, the PR | expand scope; merge; delete branches |
| **code-reviewer** | opus | full standards, the PR diff | the review (saved as `04-review.md`) | edit code; rely on the digest |
| **qa-lead** | opus | spec, design, PR diff | `05-qa-plan.md` | drive a browser; edit code |
| **qa-runner** | sonnet | the QA plan, config env facts | `05-qa-report.md` | edit code; mark unobserved scenarios Pass |

Plus the **orchestrator** вҖ” the `/feature` command running in your main
session. It launches agents, brokers their questions to you, runs the gates,
publishes the mockup Artifact, updates `STATUS.md` (it is the *only* writer),
and executes the post-merge pass. It never plans, designs, or writes code.

**The purity rule:** subagents read files, write files, and return text вҖ”
nothing interactive. Two documented exceptions: the ux-designer may invoke
the project's configured design Skill, and the qa-runner drives the
Playwright browser (that *is* its job).

## 4. The ticket

Every run lives in `docs/tickets/{id}/` вҖ” committed to git as the audit trail:

```
task.md              the raw request
STATUS.md            state machine: lane, phase, log, artifacts, cost
ROADMAP.md           only when the work was split into phases
00-intake.md         clarifying Q&A + recorded assumptions
01-spec.md           what to build + acceptance criteria
02-design.md         the design spec           в”җ Full lane
design/mockup.html   the approvable mockup     в”ҳ
03-plan.md           implementation plan (its file list = the scope contract)
standards-digest.md  just the rules this ticket touches
04-review.md         review findings + verdict (+ accepted/deferred log)
05-qa-plan.md        written UAT scenarios
05-qa-report.md      live results with evidence
```

Each artifact is written by exactly one role and consumed downstream. Later
roles never rewrite earlier roles' artifacts; answers and revisions are
appended by the orchestrator or by the owning role.

**Phase notes**

- **SCOPING** вҖ” the product-owner first checks the request isn't *already
  implemented* (cheapest possible outcome), then recommends a lane, and
  splits oversized work into a `ROADMAP.md` of sequential phases.
- **PLANNING** вҖ” the tech-lead produces the plan *and* the standards-digest,
  the distilled rule set that keeps downstream context cheap. The plan's file
  list is a contract: the reviewer flags any out-of-plan change.
- **IMPLEMENTING** вҖ” the senior-dev works in an isolated git worktree
  (`.claude/worktrees/{id}`), reused idempotently if a previous run was
  interrupted. It syncs with the default branch before the PR and on every
  fix-loop push, and must pass typecheck, lint, tests, **and build**.
- **REVIEWING** вҖ” code review and QA-plan writing run in parallel (both are
  independent reads of the same PR).
- **READY_TO_MERGE вҶ’ MERGED** вҖ” merging is yours. Afterward the orchestrator
  runs the post-merge pass: worktree/branch cleanup, a learning-capture
  proposal (a вүӨ1-paragraph standards amendment, only with your approval), and
  the offer to open the next ROADMAP phase.

**Terminal phases** are `MERGED` and `CLOSED` (with a recorded reason).
Everything else вҖ” including `READY_TO_MERGE` вҖ” is resumable.

## 6. Gates and lanes

Gates are where a human decides. Each gate's interactivity is set per lane in
the config's **gate policy table**:

- **`ask`** вҖ” you're prompted; the pipeline blocks.
- **`auto`** вҖ” the agent's recommendation is accepted and *logged*; you're
  told in the next message. An agent's real question (`NEEDS_INPUT`) always
  overrides `auto`.
- **`folded`** вҖ” the decision is batched into a later gate (e.g. Lite shows
  spec + plan together at the plan gate).

| Gate | Protects | Full | Lite | Hotfix |
|---|---|---|---|---|
| 1 lane/split | wasted ceremony | autoВ№ | autoВ№ | вҖ” |
| 2 intake | requirements | ask | ask | вҖ” |
| 3 spec | building the wrong thing | ask | foldedвҶ’5 | вҖ” |
| 4 mockup | UI direction | ask | вҖ” | вҖ” |
| 5 plan | hard-to-reverse decisions | **ask, always** | **ask** | **ask** |
| 6 merge | production | ask | ask | ask |
| 7 learning | standards drift | askВІ | askВІ | askВІ |

В№ a ROADMAP split always asks В· ВІ only when there's a proposal

Typical interaction cost: **Full вүҲ 4 prompts, Lite вүҲ 2, Hotfix вүҲ 1** вҖ” plus
the merge. If a notification mechanism is configured, the orchestrator pings
you when a gate needs you, so you never babysit the terminal.

Why this is safe: the **questioning threshold** (protocol В§5) obliges every
agent to stop and ask on *what-to-build* ambiguity and *hard-to-reverse*
technical decisions, regardless of gate policy. Auto-gates remove
rubber-stamps, not questions.

## 7. Loops вҖ” all bounded

Every loop runs at most **3 iterations**, then escalates to you:

| Loop | Trigger | Who acts |
|---|---|---|
| Intake | PO needs answers | you вҶ’ PO continues |
| Revision (spec/design/plan) | you request changes at a gate | the owning agent revises in place |
| Re-plan | senior-dev returns `PLAN_ISSUE` | tech-lead revises **aware of the worktree's partial state**; plan gate re-runs |
| Review fix | findings in `04-review.md` | senior-dev fixes вҶ’ reviewer does a **delta re-review** (new diff + prior findings only) |
| QA fix | failures in `05-qa-report.md` | senior-dev fixes вҶ’ qa-runner re-runs **only failed/blocked scenarios** |

Fix-loop policy: the default is to fix **every** finding вҖ” blocker, warning,
nit. A finding is skipped only when genuinely no-value (purely subjective, an
out-of-scope project-wide change, or a false premise), and every skip is
recorded with a reason. Real judgment calls are brokered to you.

## 8. Resumability

`STATUS.md` is the single source of truth. The orchestrator updates it after
every transition, so at any moment вҖ” after a crash, a session reload, or a
week away вҖ” a bare `/feature` finds the in-progress ticket, reads its phase,
and continues from exactly the right step. Agents are stateless between
invocations; everything they need is in the ticket folder.

This is also why **session reloads are safe**: newly installed agents only
register at session start, and the state machine doesn't care when you
restart.

## 9. Cost model

Multi-agent pipelines are expensive вҖ” industry measurements put multi-agent
systems at roughly **15Г— the tokens of a plain chat session**. The factory
controls this by design:

- **Scoped reading** вҖ” each role reads only its mapped standards sections;
  downstream agents read the per-ticket digest, not the full standards doc.
- **Model right-sizing** вҖ” judgment-heavy roles run opus; mechanical roles
  (qa-runner) and sizing roles (product-owner) run sonnet, per the config.
- **Lane discipline** вҖ” small work takes the cheap path. The 15Г— is for real
  features, where it buys spec, design, plan, review, and live QA.
- **Delta re-work** вҖ” fix-loop re-reviews and QA re-runs cover only what
  changed or failed, never the full surface again.
- **Measurement** вҖ” the orchestrator logs per-ticket cost estimates into
  `STATUS.md` (via ccusage). For per-agent dashboards, enable Claude Code's
  OpenTelemetry export with `OTEL_LOG_TOOL_DETAILS=1`.

## 10. Standardized git output

`factory/conventions.md` defines the commit format (Conventional Commits)
and the PR-description template. The senior-dev reads it directly; the
`commit` and `create-pr` skills apply the *same file* to work done outside
the factory вҖ” so all history and every PR in the project looks the same,
whoever authored it. The `factory-docs` skill generates or audits the
project's standards doc into the section structure the factory consumes.
